Title: Advanced File Encryption Using Hybrid of Modified Caesar and Hill Cipher

Overview: This project presents a secure, multi-level encryption system combining a modified version of the Caesar cipher with the Hill cipher. The system is designed to secure files during transmission, ensuring confidentiality and protection from brute-force and known-plaintext attacks. The hybrid method strengthens the encryption by leveraging both substitution and matrix-based encryption techniques.

Technologies:

  • Cryptography: Modified Caesar Cipher, Hill Cipher
  • Mathematics: Matrix multiplication for encryption and decryption
  • Programming: Python, NumPy(for matrix operations)

Key Components:

  • Hybrid Encryption System:
    Modified Caesar Cipher: Introduced dynamic key shifts based on character index to make the cipher more secure against frequency analysis and brute-force attacks.
    Hill Cipher: Applied matrix-based encryption to enhance security, particularly for large datasets and files.
  • Multi-Level Encryption:
    Files are encrypted in two stages: first with a modified Caesar cipher and then with the Hill cipher, ensuring robust security.
  • Decryption Process:
    The system also includes a two-step decryption process, reversing both the Hill and Caesar ciphers to recover the original data.

Outcome:

  • Enhanced File Security: The multi-level encryption approach makes it significantly harder for attackers to crack the system using brute force or known-plaintext attacks.
  • Secure Data Transmission: Ideal for industries requiring secure file transfers, such as legal, healthcare, and corporate sectors.
  • Improved Cryptographic Strength: By combining two encryption techniques, the system overcomes the individual weaknesses of traditional Caesar and Hill ciphers.

Development

Encryption Framework:

  • Hybrid Approach:
    The system proposes a two-layer encryption mechanism. First, it uses a modified version of the Caesar cipher for the initial level of encryption, followed by the Hill cipher for additional security. This hybrid approach ensures that even if one cipher is compromised, the data remains secure through the second layer.
  • File Encryption:
    The focus of the project is on encrypting data files to ensure secure transmission over networks, using a robust multi-level encryption approach.

Programming Languages:

  • Python: The system may have been developed using programming languages such as Python to implement cryptographic algorithms. The use of matrix operations for Hill Cipher encryption/decryption implies the use of efficient numerical libraries.

Cryptographic Algorithms

Modified Caesar Cipher:
Dynamic Key: The modification to the traditional Caesar cipher is that the key used for character substitution is dynamic rather than static. Characters at even indices are shifted forward (plus one), and characters at odd indices are shifted backward (minus one). This ensures that even repeated characters in the plaintext will have different cipher representations.
Substitution Rules: The modified Caesar cipher substitutes characters based on their index, ensuring that brute-force attacks are less effective compared to the traditional Caesar cipher.

Hill Cipher:
Matrix-Based Encryption: The Hill cipher uses matrix multiplication for encryption. The plaintext is converted into a matrix, and then multiplied by a key matrix to generate the ciphertext.
Inversibility: Decryption is possible only if the key matrix is invertible, which adds complexity and security to the encryption. The system uses a 3x3 or 6x6 matrix to enhance security by increasing the key space, making brute-force attacks impractical.

Encryption and Decryption Process

Encryption Workflow:
Level 1 (Modified Caesar Cipher): The file’s content is first encrypted using a modified Caesar cipher, where character substitution depends on the index (even or odd).
Level 2 (Hill Cipher): The partially encrypted data from the first step is then passed through the Hill cipher, which performs matrix-based encryption to add another layer of security.
File Output: The resulting ciphertext is written back into the file for secure transmission.

Decryption Workflow:
Level 1 (Hill Cipher Decryption): The first step reverses the Hill cipher operation using matrix inversion.
Level 2 (Caesar Cipher Decryption): The modified Caesar cipher is then reversed by applying the inverse operations—subtracting one from even-indexed characters and adding one to odd-indexed characters.
Final Output: The original plaintext is restored after both decryption processes.

Security Analysis

Advantages:
Resistance to Brute Force: The hybrid nature of the algorithm makes it much more resistant to brute-force attacks. The modified Caesar cipher introduces dynamic key shifts, making traditional frequency analysis difficult.
Enhanced Key Space: The Hill cipher, especially with larger matrices, provides a vast key space, making brute force attacks computationally infeasible.
Multi-Level Encryption: Using two distinct cryptographic methods (Caesar and Hill) ensures that even if one encryption layer is compromised, the data remains secure due to the second layer.

Potential Weaknesses:
Hill Cipher Key Inversibility: One limitation of the Hill cipher is that not all key matrices are invertible, which can cause issues during decryption if an invalid key is selected.

Use Cases and Applications

File Transfer Security:
The encryption system is designed for secure file transfer, protecting sensitive information during transmission over unsecured networks. This is crucial for industries dealing with confidential documents, such as legal, healthcare, and corporate sectors.

Network Security:
By securing the files transmitted over the internet, this encryption method enhances overall network security, ensuring that data is protected against unauthorized access.

Title: Data Leakage Detection in Large Language Models (LLMs)

Overview: Developed a detection pipeline to identify data leakage scenarios in Large Language Models (LLMs). The project evaluated four key leakage scenarios—Seen Q&A, Seen Question, Similar Question, and Seen Relevant Info—using multiple datasets and models. Results highlight the importance of robust benchmarks and training transparency for LLM evaluations.

Technologies:

  • Programming: Python, TensorFlow, NumPy
  • Libraries: Stanza, POS tagging, Entailment analysis
  • Models: GPT-4o, GPT-3.5 turbo, Claude-3.5 sonnet, Claude-3 sonnet, Llama 3.2

Key Components:

  • Pipeline Development: Created a stepwise detection pipeline inspired by human reasoning patterns to uncover data leakage in LLMs.
  • Evaluation Benchmarks: Designed custom benchmarks using datasets like TruthfulQA, CommonSenseQA, MMLU, and GSM8k.
  • Data Leakage Scenarios: Addressed Seen Q&A, Seen Question, Similar Question, and Relevant Info leakage, each representing distinct types of overlap between training and evaluation data.

Motivation

The project was motivated by the challenge of differentiating reasoning-based answers from memorized responses in LLMs. By mimicking human reasoning, the pipeline highlights gaps in model generalization, emphasizing the need for transparent training processes and dynamic benchmarks.

Development

Pipeline Architecture:

  • Implemented a modular pipeline to detect data leakage, separating the logic for each type of leakage scenario (Seen Q&A, Seen Q, Similar Q, and Relevant Info).
  • Leveraged Python-based frameworks like TensorFlow and Stanza for handling entailment, POS tagging, and data preprocessing.
  • Integrated entailment checks using custom prompts to evaluate model-generated outputs for relevancy and accuracy.

Automation:

  • Automated dataset sampling, prompting, and evaluation for reproducibility across multiple datasets and models.
  • Designed a judge model subsystem using GPT-4o to classify entailment in the “Relevant Info” stage of the pipeline.

Optimization:

  • Streamlined the pipeline for scalability to accommodate larger datasets and additional models in future iterations.
  • Minimized manual interventions by scripting evaluation metrics and leakage detection across all pipeline stages.

Methodology

The pipeline was developed to classify data leakage scenarios in a four-stage process:

  • Seen Q&A: Identifies cases where models directly retrieve memorized question-answer pairs.
  • Seen Question: Detects questions encountered during training without explicit answers.
  • Similar Question: Highlights reliance on semantically similar questions from training data.
  • Relevant Info: Evaluates if answers can be derived from context generated by the model.

  • TruthfulQA: Tests factual accuracy and truthfulness.
  • CommonSenseQA: Evaluates commonsense reasoning.
  • MMLU: Covers multitask language understanding.

  • Llama 3.2 (Meta): Open-source model with 3 billion parameters.

Results

Key Findings:

  • Larger models like GPT-4o and Claude-3.5 exhibit higher leakage in factual datasets like TruthfulQA.
  • Smaller models, such as Llama 3.2, show minimal leakage, reflecting better generalization.
  • TruthfulQA consistently demonstrated high leakage, while reasoning-intensive datasets like GSM8k showed reduced contamination.

Outcome: The pipeline provided a comprehensive evaluation of data leakage across models, offering insights for improving LLM evaluation practices.
